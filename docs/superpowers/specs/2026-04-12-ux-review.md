# TabLedger UI/UX Review

**Date:** 2026-04-12  
**Scope:** Full dashboard audit following Plan B (Guided Workflow) implementation

---

## Critical

### Library view inherits the full workflow header

When `state.activeView = "library"`, the phase strip ("1 Capture → 2 Review → 3 Save"), session-row, toolbar-row, filter input, and summary stats (tabs / categories / skipped) remain visible. These controls exist to serve the capture workflow only. The Library is a peer-level component, not a subview — but the current layout makes it feel like a drawer inside the workflow.

**Fix:** Hide the entire `.controls-panel` when `activeView = "library"`. The Library already has its own header, its own filter (`#archive-filter`), and its own summary (`#archive-summary`) — it doesn't need the workflow chrome.

**Location:** `styles.css` — add an `activeView`-aware rule; `dashboard.js` — write `data-active-view` on the shell in `render()` (same pattern as `data-phase`).

---

## Important

### "← Back to Review" appears in the Review phase

The CSS only hides `#back-to-review` in the Capture phase. It should only be visible in the Save phase — it has no meaning in Review.

**Fix:** Add `#back-to-review` to the Review-phase hide list in `styles.css`.

**Location:** `styles.css` lines ~1172–1178 (Review-phase block).

---

### "Skipped" count is unexplained

The summary strip shows "0 skipped" with no tooltip or label clarifying what "skipped" means (duplicate URLs? unsupported tab types?). First-time users have no context.

**Fix:** Add a `title` attribute to the skipped count element, or add a small inline clarification: "skipped (duplicates/unsupported)".

**Location:** `dashboard.html` summary strip; `dashboard.js` skipped count render.

---

### Post-save transition is abrupt

After saving, the draft disappears, the view switches to Library, and the phase strip silently resets to Capture — simultaneously, with only a small status message as feedback. Users may not realize the save succeeded, or may be disoriented by the sudden layout change.

**Fix:** Scroll the newly saved session into view in the Library and briefly highlight it. Alternatively, show a more prominent success confirmation before the transition.

**Location:** `dashboard.js` `handleSaveBookmarks()` — after `render()`, find and highlight the session element matching the saved session name.

---

### Category rename has no affordance

The Save/Cancel buttons only appear after clicking the category name input. There is no pencil icon, "click to edit" hint, or any visual signal that the name is editable. The rename feature is effectively hidden.

**Fix:** Add a small pencil icon (or `✎` glyph) next to each category name that appears on hover, making the edit affordance discoverable.

**Location:** `#category-template` in `dashboard.html`; `styles.css` category header styles.

---

## Minor

### Completion dots are color-only

Gray/amber/teal conveys status, but the only text equivalent is a `title` tooltip — invisible on touch and ignored by screen readers. A screen reader visiting a tab card cannot determine its completion state.

**Fix:** Add an `aria-label` to each `.tab-card` that includes the completion state (e.g., `"GitHub - AI filled"`), or add a visually-hidden `<span>` inside the dot with the state text.

**Location:** `dashboard.js` `renderTabItem()` — where `data-completion` is set.

---

### View tabs lack descriptive aria-labels

The Draft and Library buttons have `role="tab"` but announce as just "Draft, tab" / "Library, tab" with no indication of what activating them does.

**Fix:** Add `aria-label="Switch to Draft view"` and `aria-label="Switch to Library view"`.

**Location:** `dashboard.html` `.view-tab-bar` buttons.

---

### "Proceed to Save →" copy is vague

The next screen is for naming and saving the session. "Name & Save →" or "Save Session →" would set better expectations for what happens next.

**Location:** `dashboard.html` `#proceed-to-save`.

---

### Filter label is generic

The field is labeled "Filter" — it only filters the draft tab list, but nothing in the label signals that. "Filter Tabs" or "Search Draft" would be clearer.

**Location:** `dashboard.html` filter label inside `.session-row`.

---

### No focus management on view switch

Clicking the Draft or Library tab does not move keyboard focus into the newly active pane. Keyboard and screen reader users lose their place after switching.

**Fix:** After `render()` in the view tab click handlers, call `.focus()` on the active pane's first interactive element or its heading.

**Location:** `dashboard.js` `viewTabDraftButton` / `viewTabLibraryButton` click handlers.

---

## Recommended Implementation Order

1. **Hide controls-panel in Library view** — biggest structural issue, root of the user-reported problem
2. **Fix "Back to Review" visibility** — one-line CSS change
3. **Explain the skipped count** — one-line HTML change
4. **Category rename affordance** — pencil icon on hover
5. **Completion dot screen reader label** — add visually-hidden text in `renderTabItem()`
6. **Post-save highlight** — scroll and briefly highlight the saved session in Library
7. **View tab aria-labels** — two-attribute HTML change
8. **Focus management on view switch** — one line per click handler
9. **Copy tweaks** — "Proceed to Save", "Filter" label
